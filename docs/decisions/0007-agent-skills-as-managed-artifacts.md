# 0007. Agent skills as managed artifacts

- **Status:** accepted
- **Date:** 2026-07-03
- **Relates to:** [0003 AGENTS.md as the source of truth, with generated adapters](0003-agents-md-source-of-truth.md), [0006 Distribution and version pinning](0006-distribution-and-versioning.md)

## Context

Ma'at's instructions tell agents *what the rules are* (update protocol, doc
structure). Some workflows, however, are *procedures* — multi-step playbooks
like "retrofit documentation onto an existing repository" — that are too long
for AGENTS.md and only needed on demand. The agent ecosystem is converging on
**skills**: self-contained markdown procedures an agent loads when a task
matches (progressive disclosure).

Three questions must be settled before shipping any skill:

1. **Where do skills live?** Vendor-native locations (e.g. `.claude/skills/`)
   are not agent-agnostic, which violates Ma'at's core premise. A neutral
   location needs a discovery mechanism that works for agents *without* a
   native skills feature.
2. **Copy or symlink** into vendor-native locations for agents that do have
   one? Symlinks promise freshness but break on Windows (require Developer
   Mode; with `core.symlinks=false` git checks them out as plain text files
   containing the target path), in some agent sandboxes, and in archives and
   Docker `COPY`.
3. **How do skills evolve with the framework?** A skill scaffolded once at
   `init` and never touched again would rot silently — the exact failure mode
   Ma'at exists to prevent for docs.

The repository already solves an isomorphic problem: adapter files
(`CLAUDE.md`, `.cursor/rules/maat.mdc`, …) are generated copies of a canonical
source, regenerated by `sync`, drift-checked by `check`, and version-stamped
per ADR 0006.

## Decision

Treat skills exactly like adapters: **managed, generated artifacts**.

### 1. Canonical location: `.maat/skills/`

Skills are written to `.maat/skills/<name>/SKILL.md`. The directory is
agent-neutral, clearly Ma'at-owned, and keeps the repository root clean.

### 2. Discovery through AGENTS.md, fan-out through adapters

A managed section in `AGENTS.md` lists each skill with a one-line trigger
description and its path ("When retrofitting docs onto an existing repo, read
`.maat/skills/retrospect/SKILL.md` and follow it"). Any agent that honors
AGENTS.md — the premise of ADR 0003 — can follow a relative link, so **no
agent needs a native skills feature** for skills to work.

For agents that *do* have native skill loading, `sync` additionally
materializes the skill into the vendor location via the existing adapter
mechanism (initially: Claude Code's `.claude/skills/`). New vendor locations
are new adapter targets; the canonical source never moves.

### 3. Copies, never symlinks

Vendor fan-out writes **copies**. The only advantage of symlinks — guaranteed
freshness — is already provided by `sync` regeneration plus `check` drift
detection, with none of the Windows/sandbox/archive portability failures. This
is the same reasoning that makes `CLAUDE.md` a generated pointer file rather
than a symlink to `AGENTS.md`.

### 4. Evolution rides the existing ratchet

Skill sources are templates **embedded in the binary**
(`internal/maat/templates/skills/`), so skill content is inherently coupled to
the binary version. Generated skills carry a version stamp (ADR 0006). The
update loop is composition of existing mechanisms:

1. The repo's `maat_version` pin (or the CI Action's pinned version) moves.
2. The new binary's `sync` regenerates skills from its embedded templates.
3. `check` fails CI on drift until the regenerated skills are committed.

Skills cannot silently rot for the same reason adapters cannot.

### 5. Managed vs. user skills

Ownership is whole-file, marked by frontmatter (`managed-by: maat`): Ma'at
regenerates its own skills and **never touches** other files under
`.maat/skills/`, so teams can add hand-written skills alongside without
conflict. Hand-edits to a managed skill are overwritten by the next `sync` —
the generated-file header says so, as with every other generated artifact.

## Consequences

- **Skills stay current by construction.** The sync/check/version-pin ratchet
  that keeps adapters fresh keeps skills fresh; no auto-updater, no network,
  deterministic — consistent with ADR 0005.
- **Every agent is served.** Native pickup where a vendor location exists,
  guaranteed AGENTS.md-reference fallback everywhere else.
- **Windows and sandboxed agents just work**, because nothing is symlinked.
- Vendor fan-out duplicates skill content in the repo (one copy per native
  location plus the canonical one). Accepted: the copies are generated,
  drift-checked, and cheap; the adapter files set the precedent.
- Whole-file ownership means a team cannot customize a managed skill in place.
  The escape hatch is copying it to a new name (unmanaged) and disabling the
  managed one — a config knob that can be added when someone actually needs it.
- The adapter target model gains a second artifact kind (skills, not just
  pointer files), which `internal/maat/config.go` must represent.

## Alternatives considered

- **Vendor-native location as canonical** (`.claude/skills/`). Rejected:
  couples Ma'at to one vendor, contradicting ADR 0003's agnosticism.
- **Symlinks from vendor locations to `.maat/skills/`.** Rejected for
  portability (Windows, sandboxes, archives); copies + drift-check give the
  same freshness guarantee.
- **Scaffold-once at `init`, never regenerate.** Rejected: skills would rot
  silently as the framework evolves — the failure mode Ma'at exists to catch.
- **Fetching skills from the network at sync time.** Rejected: violates the
  deterministic, offline, single-binary model (ADR 0005/0006). Embedding in
  the binary gives the same coupling with zero moving parts.
- **A managed block inside user-editable skill files.** Rejected: whole-file
  ownership is simpler, matches the adapter precedent, and avoids merge
  ambiguity inside a file two parties edit.
